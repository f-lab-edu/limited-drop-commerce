# 패키지 구조

## 전략: 도메인(Domain) 중심 구조

기능별 레이어(`controller`, `service`, `repository`)로 나누는 대신, **도메인 단위로 묶는 구조**를 채택합니다.
규모가 커질 때 모듈 분리가 쉽고, 응집도가 높습니다.

---

## 단계 1 — 최상위 구조

```
com.mist.commerce
├── CommerceApplication.java       ← @SpringBootApplication (루트에 위치 필수)
│
├── common/                        ← 전역 공통 설정
├── config/                        ← Spring 설정 클래스
│
├── domain/                        ← 핵심 비즈니스 도메인
│   ├── user/
│   ├── product/
│   ├── order/
│   ├── reservation/
│   └── payment/
│
└── infra/                         ← 외부 연동 (PG사, 알림, 파일 등)
```

> **Spring Boot 공식 권고:** `@SpringBootApplication`이 있는 메인 클래스는 반드시 루트 패키지에 위치해야
> 컴포넌트 스캔이 전체 도메인에 적용됩니다.

---

## 단계 2 — 도메인 내부 레이어 구조

각 도메인은 동일한 내부 구조를 가집니다. `order` 도메인 예시:

```
domain/order/
├── controller/
│   └── OrderController.java          ← REST API 진입점
│
├── service/
│   ├── OrderService.java             ← 비즈니스 로직 (인터페이스)
│   └── OrderServiceImpl.java
│
├── repository/
│   └── OrderRepository.java          ← JPA Repository
│
├── entity/
│   └── Order.java                    ← @Entity
│
├── dto/
│   ├── OrderCreateRequest.java
│   ├── OrderResponse.java
│   └── OrderStatusUpdateRequest.java
│
└── exception/
    ├── OrderNotFoundException.java
    └── OrderAlreadyCanceledException.java
```

---

## 단계 3 — 핵심 도메인 관계 정의

```
user       ─┐
product    ─┤─→  reservation ─→  order ─→  payment
             └───────────────────────────────────────
```

| 도메인 | 역할 |
|--------|------|
| `user` | 회원 가입/인증, Security 연동 |
| `product` | 상품 정보, 재고 관리 |
| `reservation` | 한정 수량 선점/예약 (대기열 포함 가능) |
| `order` | 예약 → 주문 확정, 주문 상태 관리 |
| `payment` | PG 연동, 결제/취소/환불 |

---

## 단계 4 — `global` 패키지

기존 `common/`과 `config/`를 `global` 패키지로 통합합니다.
도메인과 무관하게 애플리케이션 전역에서 동작하는 **횡단 관심사(Cross-Cutting Concerns)** 를 한 곳에서 관리합니다.

```
global/
├── config/                              ← Spring 설정 클래스
│   ├── SecurityConfig.java              ← Spring Security 설정
│   ├── JpaConfig.java                   ← JPA Auditing 활성화 (@EnableJpaAuditing)
│   └── WebMvcConfig.java                ← CORS, Interceptor 등록
│
├── exception/                           ← 전역 예외 처리
│   ├── GlobalExceptionHandler.java      ← @RestControllerAdvice
│   ├── BusinessException.java           ← 비즈니스 예외 최상위 클래스
│   └── ErrorDetail.java                 ← 검증 오류 상세 DTO
│
├── response/                            ← 공통 API 응답 포맷
│   └── ApiResponse.java                 ← ApiResponse<T>
│
├── entity/                              ← 공통 엔티티 기반 클래스
│   └── BaseTimeEntity.java              ← createdAt, updatedAt (@MappedSuperclass)
│
├── filter/                              ← Servlet Filter (DispatcherServlet 이전)
│   └── MDCFilter.java                   ← requestId, userId를 MDC에 주입
│
├── interceptor/                         ← Spring MVC Interceptor (DispatcherServlet 이후)
│   └── LoggingInterceptor.java          ← 요청/응답 로깅
│
└── util/                                ← 범용 유틸리티
    └── SecurityUtils.java               ← 인증 사용자 정보 조회
```

### 패키지별 역할 요약

| 패키지 | 역할 | 주요 클래스 |
|--------|------|-------------|
| `config/` | Spring Bean 설정 및 기능 활성화 | `SecurityConfig`, `JpaConfig`, `WebMvcConfig` |
| `exception/` | 전역 예외 포착 및 일관된 에러 응답 변환 | `GlobalExceptionHandler`, `BusinessException` |
| `response/` | 모든 API의 응답 구조 표준화 | `ApiResponse<T>` |
| `entity/` | JPA 엔티티 공통 필드 상속 | `BaseTimeEntity` |
| `filter/` | Servlet 레이어 전처리 (MDC 주입, 인증 토큰 파싱) | `MDCFilter` |
| `interceptor/` | Spring MVC 레이어 전처리 (로깅) | `LoggingInterceptor` |
| `util/` | 도메인에 속하지 않는 공통 유틸 | `SecurityUtils` |

### filter vs interceptor 분리 기준

```
HTTP 요청
   ↓
[Servlet Filter]       ← global/filter/       모든 요청, DispatcherServlet 이전
   ↓
DispatcherServlet
   ↓
[MVC Interceptor]      ← global/interceptor/  Spring MVC 요청만, Controller 이전
   ↓
Controller
```

| 구분 | 레이어 | 사용 시점 |
|------|--------|-----------|
| `Filter` | Servlet | MDC 주입, 인증 토큰 파싱 — 모든 요청에 가장 먼저 적용 |
| `Interceptor` | Spring MVC | 로깅, 권한 체크 — Controller 진입 직전/직후 |

### `global` vs `domain` 분리 기준

| 위치 | 기준 | 예시 |
|------|------|------|
| `global/exception/` | 전역 핸들러, 공통 예외 기반 클래스 | `BusinessException`, `GlobalExceptionHandler` |
| `domain/{name}/exception/` | 도메인 전용 예외 | `ProductNotFoundException`, `StockExhaustedException` |
| `global/entity/` | 모든 엔티티가 상속하는 공통 필드 | `BaseTimeEntity` |
| `domain/{name}/entity/` | 도메인 전용 엔티티 | `Product`, `Order` |

> 두 개 이상의 도메인에서 공유하거나 도메인과 무관하게 동작하는 코드는 `global`에 둔다.

## 단계 5 — `infra` (외부 연동)

```
infra/
├── payment/
│   └── TossPaymentClient.java        ← PG사 HTTP 클라이언트
├── notification/
│   └── EmailNotificationSender.java
└── storage/
    └── S3FileUploader.java
```

> `infra`는 외부 시스템 의존성을 격리합니다. PG사를 교체해도 `domain/payment`의 비즈니스 로직은 변경 불필요.

```
│   ├── exception/

│   │   └── ErrorDetail.java
│   ├── response/

│   ├── entity/
│   │   └── BaseTimeEntity.java
│   ├── filter/
│   │   └── MDCFilter.java
│   ├── interceptor/
│   │   └── LoggingInterceptor.java
│   └── util/
│       └── SecurityUtils.java
│
├── domain/
│   ├── user/
│   │   └── controller/ service/ repository/ entity/ dto/ exception/
│   ├── product/
│   │   └── controller/ service/ repository/ entity/ dto/ exception/
│   ├── reservation/
│   │   └── controller/ service/ repository/ entity/ dto/ exception/
│   ├── order/
│   │   └── controller/ service/ repository/ entity/ dto/ exception/
│   └── payment/
│       └── controller/ service/ repository/ entity/ dto/ exception/
│
└── infra/
    ├── payment/
    ├── notification/
    └── storage/
```
