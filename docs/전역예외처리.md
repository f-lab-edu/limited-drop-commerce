# 전역 예외 처리 설계 가이드

## 1. 정의 및 원칙

- 애플리케이션 전역에서 발생하는 예외는 `@RestControllerAdvice`를 통해 일관된 형식으로 처리한다.
- 예외 응답은 공통 JSON 구조를 사용하며, 상태 코드와 에러 코드를 함께 반환한다.
- Validation 예외, 비즈니스 예외, 인증/인가 예외, 시스템 예외를 구분하여 처리한다.
- 서버 내부 구현 정보와 상세 스택 트레이스는 외부 응답에 노출하지 않는다.
- 처리되지 않은 예외는 500 Internal Server Error로 변환하고, 상세 원인은 서버 로그에 기록한다.
- Spring Security 및 Filter 레벨 예외는 필요 시 별도 Handler와 함께 운영한다.

## 2. 예외 계층 구조

```
Exception
└── RuntimeException
    └── BusinessException                  ← 비즈니스 규칙 위반 (4xx 계열)
        ├── EntityNotFoundException        ← 리소스 없음 (404)
        ├── StockExhaustedException        ← 재고 부족 (400)
        ├── DuplicateOrderException        ← 중복 주문 (409)
        └── PaymentFailedException         ← 결제 실패 (400)
```

- **`BusinessException`** : HTTP Status + 에러 코드를 함께 보유하는 최상위 비즈니스 예외
- 도메인별 예외는 `BusinessException`을 상속해 의미를 명확히 한다
- 인프라/외부 연동 예외는 `BusinessException`으로 변환하거나 별도 처리한다

### BusinessException 구현

```java
@Getter
public class BusinessException extends RuntimeException {

    private final String code;
    private final HttpStatus httpStatus;

    public BusinessException(String code, String message, HttpStatus httpStatus) {
        super(message);
        this.code = code;
        this.httpStatus = httpStatus;
    }
}
```

### 도메인별 예외 예시

```java
public class EntityNotFoundException extends BusinessException {
    public EntityNotFoundException(String entityName, Long id) {
        super(entityName.toUpperCase() + "_NOT_FOUND",
              entityName + "을(를) 찾을 수 없습니다. (id: " + id + ")",
              HttpStatus.NOT_FOUND);
    }
}

public class StockExhaustedException extends BusinessException {
    public StockExhaustedException() {
        super("STOCK_EXHAUSTED", "재고가 부족합니다.", HttpStatus.BAD_REQUEST);
    }
}

public class DuplicateOrderException extends BusinessException {
    public DuplicateOrderException() {
        super("DUPLICATE_ORDER", "이미 처리 중인 주문이 있습니다.", HttpStatus.CONFLICT);
    }
}
```

## 3. 에러 코드 컨벤션

| 유형 | 코드 형식 | 예시 |
|------|-----------|------|
| 공통 | `COMMON_{의미}` | `COMMON_INVALID_INPUT` |
| Validation | `VALIDATION_ERROR` | 고정 사용 |
| 도메인 | `{도메인}_{의미}` | `PRODUCT_NOT_FOUND`, `ORDER_ALREADY_CANCELED` |
| 인증/인가 | `AUTH_{의미}` | `AUTH_UNAUTHORIZED`, `AUTH_ACCESS_DENIED` |
| 시스템 | `INTERNAL_SERVER_ERROR` | 고정 사용 |

- 형식: **SCREAMING_SNAKE_CASE**
- 클라이언트는 `message`가 아닌 `code`를 기준으로 분기 처리한다

## 4. 예외 처리 범위 및 우선순위

| 우선순위 | 예외 타입 | HTTP Status | 에러 코드 |
|----------|-----------|-------------|-----------|
| 1 | `MethodArgumentNotValidException` | 400 | `VALIDATION_ERROR` |
| 2 | `ConstraintViolationException` | 400 | `VALIDATION_ERROR` |
| 3 | `HttpMessageNotReadableException` | 400 | `COMMON_INVALID_INPUT` |
| 4 | `BusinessException` (하위 포함) | 도메인 정의 | 도메인 정의 |
| 5 | `AccessDeniedException` | 403 | `AUTH_ACCESS_DENIED` |
| 6 | `AuthenticationException` | 401 | `AUTH_UNAUTHORIZED` |
| 7 | `Exception` (catch-all) | 500 | `INTERNAL_SERVER_ERROR` |

> `@ExceptionHandler`는 가장 구체적인 예외 타입부터 처리된다. catch-all `Exception`은 반드시 마지막에 위치시킨다.

## 6. Spring Security Filter 레벨 예외 처리

`@RestControllerAdvice`는 DispatcherServlet 이후에 동작하므로, **Security Filter 단계에서 발생한 예외는 잡지 못한다**.
인증(401) · 인가(403) 예외는 별도 핸들러로 처리한다.

## 7. 로깅 전략

예외 심각도에 따라 로그 레벨을 구분한다.

| 예외 유형 | 로그 레벨 | 이유 |
|-----------|-----------|------|
| Validation 실패 | 로그 없음 | 정상적인 클라이언트 오류 |
| BusinessException | `WARN` | 비즈니스 흐름 이탈, 모니터링 필요 |
| 인증/인가 실패 | `WARN` | 보안 감사 필요 |
| 미처리 예외 (500) | `ERROR` + 스택 트레이스 | 즉시 확인 필요 |

```java
// BusinessException: 스택 트레이스 없이 핵심 정보만 기록
log.warn("[BusinessException] code={}, message={}", ex.getCode(), ex.getMessage());

// 미처리 예외: 전체 스택 트레이스 기록
log.error("[UnhandledException] {}", ex.getMessage(), ex);
```

- MDC에 `requestId`, `userId` 등을 등록해 로그 추적이 가능하도록 한다.
- 500 에러는 알림(Slack, PagerDuty 등)과 연동해 즉시 인지할 수 있도록 구성한다.

## 8. ErrorDetail (검증 오류 상세)

```java
@Getter
@AllArgsConstructor
public class ErrorDetail {
    private final String field;
    private final Object value;
    private final String reason;

    public static ErrorDetail of(FieldError error) {
        return new ErrorDetail(
            error.getField(),
            error.getRejectedValue(),
            error.getDefaultMessage()
        );
    }
}
```

## 9. 주의사항

- `@ExceptionHandler(Exception.class)` catch-all은 반드시 존재해야 한다. 없으면 미처리 예외가 Spring 기본 에러 응답(`/error`)으로 처리되어 응답 포맷이 깨진다.
- `ResponseEntityExceptionHandler`를 상속하면 Spring MVC 내장 예외(`MethodArgumentNotValidException` 등)를 오버라이드할 수 있다. 상속하지 않으면 내장 예외가 기본 핸들러와 충돌할 수 있다.
- `BusinessException`을 `catch (Exception e)`로 잡아 500으로 올리는 실수를 방지하기 위해, `@ExceptionHandler` 순서를 항상 구체적 타입 → 범용 타입 순으로 정의한다.
- 외부 API 연동 실패(타임아웃, 서킷브레이커 등)는 `BusinessException`으로 변환하거나 별도 핸들러를 추가해 일관된 포맷으로 응답한다.
