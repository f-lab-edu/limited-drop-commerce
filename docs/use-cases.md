## UC-01: 일반 회원가입

| 항목 | 내용 |
|---|---|
| **액터** | 고객 |
| **목적** | 드롭 이벤트 참여를 위한 일반 계정 생성 |
| **사전 조건** | 없음 |

**기본 흐름**

1. 고객이 이메일, 비밀번호, 이름을 입력한다.
2. 시스템이 이메일 중복 여부를 확인한다.
3. 시스템이 비밀번호를 암호화하여 계정을 저장한다.
4. 회원가입이 완료된다.

**예외 흐름**

- E1. 이미 등록된 이메일 → `USER_EMAIL_DUPLICATED` 에러 반환
- E2. 이메일 형식이 올바르지 않은 경우 → `VALIDATION_ERROR` 에러 반환

**사후 조건**: 고객 계정이 생성되고 로그인 가능 상태가 된다.

## UC-02: 기업 회원가입

| 항목 | 내용 |
|---|---|
| **액터** | 기업 담당자 |
| **목적** | 서비스 이용을 위한 기업 계정 및 담당자 계정 생성 |
| **사전 조건** | 없음 |

**기본 흐름**

1. 기업 담당자가 기업 회원가입 화면에 진입한다.
2. 회사명, 사업자등록번호, 대표 연락처, 주소 등 기업 정보를 입력한다.
3. 시스템이 사업자등록번호 유효성을 검증한다.
4. 시스템이 담당자 이메일 중복 여부를 확인한다.
5. 시스템이 기업 정보와 담당자 계정을 생성한다.

**예외 흐름**

- E1. 사업자등록번호가 유효하지 않으면 가입을 중단한다.
- E2. 이미 등록된 사업자등록번호이면 중복 메시지를 반환한다.
- E3. 담당자 이메일이 중복되면 계정을 생성하지 않는다.

**사후 조건**: 기업 정보와 회원 계정이 생성되고 로그인 가능 상태가 된다.

## UC-03: 로그인

| 항목 | 내용 |
|---|---|
| **액터** | 회원 (일반 또는 기업) |
| **목적** | 등록된 계정으로 서비스에 로그인 |
| **사전 조건** | 등록된 계정이 존재함 |

**기본 흐름**

1. 회원이 이메일과 비밀번호를 입력한다.
2. 시스템이 계정 존재 여부를 확인한다.
3. 시스템이 비밀번호를 검증한다.
4. 시스템이 계정 상태(활성/잠금)를 확인한다.
5. 시스템이 인증 정보(JWT 토큰)를 발급한다.
6. 로그인에 성공하고 메인 페이지로 이동한다.

**예외 흐름**

- E1. 이메일 또는 비밀번호가 일치하지 않으면 로그인 실패 메시지를 반환한다.
- E2. 로그인 실패 횟수 초과 시 계정을 잠근다.

**사후 조건**: 회원이 인증되어 보호된 리소스에 접근할 수 있다.

## UC-06: 재고 선점 (선착순)

| 항목 | 내용 |
|---|---|
| **액터** | 고객 |
| **목적** | 드롭 이벤트 오픈 시 한정 재고를 선착순으로 선점 |
| **사전 조건** | 로그인 상태, 드롭 이벤트가 `OPEN` 상태 |

**기본 흐름**

1. 고객이 원하는 상품 옵션과 수량을 선택하고 선점을 요청한다.
2. 시스템이 Redis 분산 락을 획득한다.
3. 시스템이 잔여 재고를 확인한다.
4. 시스템이 Redis에 재고를 차감하고 예약 정보를 생성한다.
5. 시스템이 분산 락을 해제한다.
6. 시스템이 예약 ID와 결제 만료 시각을 반환한다.

**예외 흐름**

- E1. 락 획득 실패(경쟁 과다) → 잠시 후 재시도 안내
- E2. 잔여 재고가 0인 경우 → `STOCK_EXHAUSTED` 에러 반환
- E3. 이벤트가 아직 오픈되지 않은 경우 → `DROP_EVENT_NOT_OPEN` 에러 반환

**사후 조건**: 예약이 `RESERVED` 상태로 생성되고 결제 대기 상태가 된다. 일정 시간 내 결제하지 않으면 만료된다.

## UC-07: 예약 취소

| 항목 | 내용 |
|---|---|
| **액터** | 고객 |
| **목적** | 결제 전 선점한 재고 예약을 직접 취소 |
| **사전 조건** | 로그인 상태, `RESERVED`(결제 대기) 상태의 예약 존재 |

**기본 흐름**

1. 고객이 본인의 예약 취소를 요청한다.
2. 시스템이 예약 상태를 `CANCELED`로 변경한다.
3. 시스템이 Redis에서 선점된 재고를 복구하고 상태를 `RELEASED`로 변경한다.
4. 취소 완료를 반환한다.

**예외 흐름**

- E1. 이미 주문이 확정된 예약인 경우 → `RESERVATION_ALREADY_CONFIRMED` 에러 반환
- E2. 본인의 예약이 아닌 경우 → `AUTH_ACCESS_DENIED` 에러 반환

**사후 조건**: 재고가 복구되어 다른 고객이 선점할 수 있다.

## UC-08: 주문 생성

| 항목 | 내용 |
|---|---|
| **액터** | 고객 |
| **목적** | 재고 선점 후 주문 확정 |
| **사전 조건** | 로그인 상태, `RESERVED` 상태의 유효한 예약 존재 |

**기본 흐름**

1. 고객이 예약 ID와 배송지 정보를 포함하여 주문을 요청한다.
2. 시스템이 예약 유효성(상태, 만료 시각)을 검증한다.
3. 시스템이 주문을 생성하고 상태를 `PENDING_PAYMENT`로 설정한다.
4. 시스템이 예약 상태를 `CONFIRMED`로 변경한다.
5. 시스템이 주문 ID와 결제 가능 금액을 반환한다.

**예외 흐름**

- E1. 예약이 만료된 경우 → `RESERVATION_EXPIRED` 에러 반환
- E2. 예약 상태가 `RESERVED`가 아닌 경우 → `RESERVATION_INVALID_STATE` 에러 반환

**사후 조건**: 주문이 `PENDING_PAYMENT` 상태로 생성된다. 결제를 완료해야 최종 확정된다.

## UC-10: 주문 취소

| 항목 | 내용 |
|---|---|
| **액터** | 고객 |
| **목적** | 결제 전 또는 결제 완료 후 주문 취소 |
| **사전 조건** | 로그인 상태, 주문 상태가 `PENDING_PAYMENT` 또는 `PAID` |

**기본 흐름**

1. 고객이 주문 취소를 요청한다.
2. 시스템이 주문 취소 가능 상태를 확인한다.
3. 주문 상태가 `PAID`인 경우 PG사에 환불을 요청한다.
4. 시스템이 주문 상태를 `CANCELED`로 변경한다.
5. 시스템이 Kafka에 `OrderCanceled` 이벤트를 발행한다.
6. Kafka Consumer가 이벤트를 처리하여 재고를 복구한다.

**예외 흐름**

- E1. 이미 배송 중인 경우 → `ORDER_CANNOT_CANCEL` 에러 반환
- E2. PG사 환불 실패 → `PAYMENT_REFUND_FAILED` 에러 반환, 수동 처리 대기

**사후 조건**: 주문이 취소되고 재고가 복구된다.

## UC-11: 결제 요청

| 항목 | 내용 |
|---|---|
| **액터** | 고객 |
| **목적** | 주문에 대한 결제 처리 |
| **사전 조건** | 로그인 상태, `PENDING_PAYMENT` 상태의 주문 존재 |

**기본 흐름**

1. 고객이 주문 ID와 결제 수단 정보를 포함하여 결제를 요청한다.
2. 시스템이 주문 금액과 결제 요청 금액의 일치 여부를 검증한다.
3. 시스템이 멱등성 키(Idempotency Key)를 생성하여 PG사에 결제를 요청한다.
4. PG사로부터 결제 승인 응답을 받는다.
5. 시스템이 주문 상태를 `PAID`로 변경한다.
6. 시스템이 Kafka에 `PaymentCompleted` 이벤트를 발행한다.

**예외 흐름**

- E1. 금액 불일치 → `PAYMENT_AMOUNT_MISMATCH` 에러 반환
- E2. PG사 결제 실패 → `PAYMENT_FAILED` 에러 반환, 주문 상태 유지
- E3. 중복 결제 요청(동일 Idempotency Key) → 이전 결제 결과를 그대로 반환

**사후 조건**: 결제가 완료되고 주문이 `PAID` 상태가 된다.

## UC-12: 결제 취소 / 환불

| 항목 | 내용 |
|---|---|
| **액터** | 고객 |

**예외 흐름**

- E1. PG사 취소 실패 → `PAYMENT_CANCEL_FAILED` 에러 반환, 관리자 수동 처리 등록

**사후 조건**: 환불이 완료되고 재고가 복구된다.

**사후 조건**: 드롭 이벤트가 등록되어 오픈 시각에 고객이 참여할 수 있다.

## UC-14: 미결제 주문 만료 처리 (시스템)

3. 시스템이 각 만료 주문에 대해 Kafka에 `OrderExpired` 이벤트를 발행한다.

**예외 흐름**

- 없음

**사후 조건**: 미결제 주문이 만료 처리되고 재고 복구 이벤트가 Kafka에 발행된다.

## UC-15: 만료/취소 주문 재고 복구 (시스템)

| 항목 | 내용 |
|---|---|
| **액터** | 시스템 (Kafka Consumer) |
| **목적** | 만료/취소된 주문의 재고를 복구하여 다른 고객이 선점할 수 있도록 함 |
| **사전 조건** | Kafka에서 `OrderExpired` 또는 `OrderCanceled` 이벤트 수신 |

**기본 흐름**

1. Kafka Consumer가 `OrderExpired` 또는 `OrderCanceled` 이벤트를 수신한다.
2. 시스템이 해당 주문의 예약 정보를 조회한다.
3. 시스템이 Redis 재고를 원복하고 예약 상태를 `EXPIRED` 또는 `CANCELED`로 변경한다.
4. 재고 복구 완료를 기록한다.

**예외 흐름**

- E1. 이벤트 처리 실패 시 Kafka 재처리(retry) 또는 DLQ(Dead Letter Queue)로 이동

**사후 조건**: 재고가 복구되어 다른 고객이 해당 수량을 선점할 수 있다.
