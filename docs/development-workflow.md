# 개발 작업 흐름

이 문서는 구현과 검증에 대한 작업 기준을 정의한다.
브랜치 전략과 커밋 규칙은 `docs/branch-rules.md`를 따른다.

---

## 작업 완료 기준

작업 완료를 보고하기 전에 변경 범위에 맞는 자동화 검증을 실행하고 결과를 확인한다.

- TDD 기반 구현은 실패하는 테스트를 먼저 작성하고, 구현 후 관련 테스트가 통과해야 완료로 본다.
- 통합 테스트가 포함된 작업은 테스트 통과만으로 완료 처리하지 않는다. 실제 local 환경에서 `compose.dev.yml`로 앱을 시작한 뒤 핵심 동작을 확인해야 한다.
- local 환경 검증은 통합 테스트 성격의 작업에만 필수로 적용한다. 단순 단위 테스트, 문서 수정, 순수 리팩터링에는 변경 위험도에 맞춰 필요한 검증만 수행한다.
- local 앱 실행은 다음 명령을 기준으로 한다.

```bash
docker compose -f compose.dev.yml up -d app
```

- 검증 완료 보고에는 실행한 테스트/명령과 확인한 local 동작 결과를 함께 적는다.

## 의존 도메인 최소 스텁 원칙

이슈의 주 기능을 구현하는 과정에서 다른 도메인이 참조 대상으로 필요한 경우, 해당 도메인은 **최소 스텁**으로만 구현한다.

### 스텁의 범위

스텁은 주 기능의 컴파일과 테스트가 통과하기 위한 최소 수준에 한정한다.

| 항목 | 스텁에 포함 | 스텁에서 제외 |
|---|---|---|
| 엔티티 | 참조에 필요한 필드, 기본 팩토리 메서드 | 도메인 규칙 검증, 상태 전이 메서드 |
| Repository | `JpaRepository<T, ID>` 선언 | 커스텀 쿼리, N+1 최적화 |
| 테스트 | 해당 없음 (작성하지 않음) | 단위 테스트, 영속성 테스트 전체 |
| DTO | 해당 없음 | 요청/응답 DTO |
| 예외 | 해당 없음 | 도메인 예외 클래스 |

### 커밋 제외

스텁 파일은 이번 이슈의 PR 커밋에 포함하지 않는다.

- `git add` 시 스텁 파일을 스테이징 목록에서 제외한다.
- 스텁이 필요한 도메인과 해당 이슈 번호를 TODO.md에 명시한다.
- 전체 구현은 해당 도메인 전용 이슈에서 진행한다.

### TODO.md 표기

스텁 섹션은 제목 뒤에 `⚠️ 커밋 제외` 를 표기하고 범위 제한 이유를 설명한다.

```markdown
### N. [도메인명] 최소 스텁 구현 ⚠️ 커밋 제외

> [이유 설명]. 전체 구현은 [이슈 번호 또는 별도 이슈]에서 진행한다.
> 이 섹션의 파일들은 이번 PR 커밋에 포함하지 않는다.
```
